## Document Style

- **Narrative over bullets**: Docs should teach a mental model, not just list requirements. Walk through the problem, explain the insight, THEN get to the solution.
- **Concrete examples**: Use real scenarios, real numbers. Abstract descriptions are harder to react to.
- **Visualize systems**: Mermaid diagrams for state machines and workflows. ASCII mockups and tables for UI concepts.
- **Explicit non-goals**: State what you're NOT building. Prevents scope creep and misaligned expectations.
- **Open questions as prompts**: End docs with questions that need stakeholder input, framed as discussion prompts.
- **Jobs to Be Done format**: When [situation] → I want [capability] → So that [outcome].
